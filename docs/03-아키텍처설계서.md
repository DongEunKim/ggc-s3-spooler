## 2. 컴포넌트 아키텍처

```
src/spooler/
├── __main__.py         ← 진입점, asyncio 루프 관리, 시그널 처리
├── config.py           ← 런타임 설정 dataclass + INI 파일 지원
├── filename_codec.py   ← 파일명 인코딩/디코딩 (순수 함수)
├── watcher.py          ← watchdog 기반 파일 감시 + 전송 루프
├── cleaner.py          ← 보존기간/용량 초과 파일 정리
├── stream_client.py    ← Stream Manager SDK 래퍼
├── client_protocol.py  ← FileTransferClient Protocol 정의
└── file_stability.py   ← 비동기 파일 안정성 검증
```

### 2.1 스레드/비동기 모델

```
Main Thread (asyncio event loop)
├── SpoolWatcher.process_loop()  ← asyncio coroutine
│     └── asyncio.Queue로부터 파일 경로 수신
│     └── run_in_executor()로 Stream Manager SDK 호출 (blocking)
├── cleaner_loop()               ← asyncio coroutine (주기적 실행)
└── SpoolEventHandler            ← watchdog 스레드에서 실행
      └── loop.call_soon_threadsafe() → asyncio Queue에 push
```

**설계 근거**: Stream Manager Python SDK는 동기 블로킹 API이다. `run_in_executor`를 사용하여 이벤트 루프를 블록하지 않으면서도 asyncio 패턴을 유지한다. watchdog은 내부적으로 별도 스레드를 사용하므로 `call_soon_threadsafe`로 이벤트 루프에 안전하게 작업을 전달한다.

## 2.2 Protocol 기반 클라이언트 추상화 (CAN Blackbox 패턴)

```
FileTransferClient (Protocol)        ← 타입 시스템 레벨 추상화 (spooler.client_protocol)
├── S3ExportUploader      ← 실제 Stream Manager SDK 연동 (Pattern 2)
├── S3SpoolerClient          ← 위를 감싼 업로드+삭제 워크플로우
├── (spooler_testing) Mock 클라이언트 ← 테스트용 완전 격리 Mock (배포 번들 제외)
└── (향후 확장)                       ← 다른 전송 백엔드 (예: HTTP upload) 지원 가능
```

> Mock 클라이언트(`MockS3SpoolerClient` 등)는 운영 패키지가 아닌
> `src/spooler_testing/`에 위치하여 배포 아티팩트에 포함되지 않는다.

**설계 근거**: CAN Blackbox 시스템에서 채택한 Protocol-based Dependency Injection 패턴을 적용하여 런타임 타입 안전성과 테스트 격리를 동시에 확보한다. Python 3.8+ `typing.Protocol`을 사용하여 Duck Typing과 정적 타입 체크를 결합한다.

**Protocol 정의** (`src/spooler/client_protocol.py` 실제 정의):
```python
@runtime_checkable
class FileTransferClient(Protocol):
    def connect(self) -> None:
        """클라이언트 연결을 초기화한다."""
        ...

    def close(self) -> None:
        """클라이언트 연결을 종료한다."""
        ...

    def append_file(self, stream_name: str, s3_key: str, file_path: Path) -> None:
        """파일을 스트림에 전송하고 전송 완료 후 파일을 삭제한다."""
        ...
```

### 2.3 하이브리드 파일 안정성 검증

```
파일 생성 감지 (watchdog)
├── 1차 필터: 시간 기반 사전 검증 (~100ms 응답)
│   └── mtime 기준 debounce 처리 (file_stability_wait)
└── 2차 검증: 크기 기반 안정성 확인
    ├── 파일 크기 추적 (stability_check_interval 간격)
    ├── 연속 stability_check_count 회 크기 불변 시 안정 판정
    └── max_stability_wait 타임아웃으로 강제 진행
```

**설계 근거**: CAN Blackbox 시스템의 실시간성 요구사항에서 착안한 하이브리드 접근법이다. 시간 기반 검증으로 대부분의 일반적 케이스를 빠르게 처리하고, 크기 기반 검증으로 느린 쓰기나 네트워크 스토리지 지연을 안정적으로 처리한다.

**성능 특성**:
- **실시간성 유지**: 일반 파일 (빠른 쓰기) → ~100ms 내 처리
- **안정성 보장**: 느린 쓰기/대용량 파일 → 크기 불변 확인 후 처리  
- **오버헤드 최소화**: 시간 기반 사전 필터링으로 불필요한 크기 체크 방지

### 2.4 INI 설정 계층 구조

```
설정 우선순위: CLI Arguments > 환경변수 > INI File > Default Values

CLI: python -m spooler --spool-dir /custom/path --config custom.ini
Env: S3_SPOOLER_* 환경변수 (예: S3_SPOOLER_S3_BUCKET)
INI: spooler.ini 파일의 섹션별 설정 (/etc/ggc-s3-spooler/ 우선)
Default: SpoolerConfig dataclass의 기본값
```

**spooler.ini 섹션 구조** (CAN Blackbox 스타일):
```ini
[spooler]
spool_dir = /var/spool/s3-spooler/spool
log_level = INFO

[stream_manager]
host = localhost
port = 8088

[cleanup]
max_spool_size_mb = 900
file_retention_hours = 120
poll_interval_seconds = 5

[stability]
file_stability_wait = 0.1
stability_check_interval = 0.2
stability_check_count = 3
max_stability_wait = 10.0
```

## 3. 데이터 흐름

### 3.1 정상 전송 흐름 (하이브리드 안정성 검증)

```
1. Client Process
   → spool_dir에 파일 생성 (atomic rename 권장)
     파일명: {stream_id}__{s3_key_encoded}

2. SpoolEventHandler (watchdog thread)
   → FileCreatedEvent 감지
   → asyncio Queue에 파일 경로 push

3. SpoolWatcher.process_loop() (asyncio coroutine)
   → Queue에서 Path 꺼냄
   → 1차 검증: 시간 기반 안정성 대기 (file_stability_wait)
   → 2차 검증: FileStabilityChecker.wait_for_stability()
     ├── stability_check_interval 간격으로 파일 크기 모니터링
     ├── 연속 stability_check_count 회 크기 불변 → 안정 판정
     └── max_stability_wait 타임아웃 → 강제 진행

4. 파일 안정성 확인 후
   → filename_codec.decode() 로 메타데이터 추출
   → FileTransferClient.append_file() 호출 (Protocol 추상화)

5. FileTransferClient 구현체 (S3SpoolerClient)
   → Stream Manager SDK: S3ExportTaskDefinition append_message

6. Stream Manager
   → S3 Export 설정에 따라 S3에 업로드

7. SpoolWatcher
   → 전송 완료 확인 후 파일 삭제
```

### 3.2 기동 시 미전송 파일 처리

```
Spooler 시작
└── SpoolWatcher._drain_existing()
    → spool_dir 스캔 (mtime 오름차순)
    → 각 스풀 파일에 대해 _transfer() 실행
```

### 3.3 자동 정리 흐름

```
cleaner_loop() — poll_interval_seconds마다 실행
└── SpoolCleaner.run_once()
    ├── _evict_expired(): mtime > retention_hours 파일 삭제
    └── _evict_over_quota(): 총 크기 > max_spool_size_mb 시
        mtime 오름차순으로 한도 이하가 될 때까지 삭제
```

## 5. 개발환경 제약 사항

현재 개발환경(로컬 Mac/Linux)은 실제 Greengrass Core Device와 다르다.  
아래 항목은 **로컬에서 검증 불가**하며, 실제 디바이스 배포 후 별도 검증이 필요하다.

| 항목 | 개발환경 | 실제 Core Device |
|------|----------|-----------------|
| Stream Manager SDK 연결 | `spooler_testing` Mock 대체 | `stream_manager` SDK 실동작 |
| Greengrass 레시피 적용 | 불가 | `greengrass-cli deployment create` |
| 컴포넌트 설정 `{configuration:/key}` | 불가 | Greengrass Nucleus가 치환 |
| IPC 소켓 / 권한 모델 | 없음 | `ggc_user` 계정, `/greengrass/v2` |
| S3 업로드 실동작 | 없음 | Stream Manager → IoT Core → S3 |

**테스트 전략**:
- 로컬: `spooler_testing` Mock 기반 단위·통합 테스트로 로직 검증
- 디바이스: `@pytest.mark.greengrass` 마크된 테스트를 실디바이스에서 실행

## 6. 향후 확장 계획 — 네트워크 파일 공급 (Phase 2)

> **전제 조건**: 로컬 파일 기반 기본 기능(FR-01~FR-05) 완전 구현 및 디바이스 검증 후 진행

### 6.1 확장 목적

코어 디바이스 하위 네트워크에 연결된 **클라이언트 디바이스**가 HTTPS 등 네트워크 수단으로 스풀러에 파일을 직접 공급할 수 있도록 한다.

### 6.2 확장 컨텍스트 다이어그램

```
        Sub-network (LAN / Greengrass sub-network)
        ┌──────────────────────────────────────────┐
        │                                          │
        │  ┌─────────────┐   HTTPS multipart       │
        │  │  Client     ├──────────────────────┐  │
        │  │  Device A   │                      │  │
        │  └─────────────┘                      │  │
        │  ┌─────────────┐   POST /upload        │  │
        │  │  Client     ├──────────────────────┤  │
        │  │  Device B   │                      ▼  │
        └──┴─────────────┴──────────────────────┼──┘
                                                │
                        ┌───────────────────────┼───────────────────────┐
                        │  Greengrass Core Device│                       │
                        │                        ▼                       │
                        │  ┌─────────────────────────────────────────┐  │
                        │  │        HTTP Intake Server (Phase 2)     │  │
                        │  │  POST /upload → spool_dir에 파일 기록   │  │
                        │  └───────────────────┬─────────────────────┘  │
                        │                      │ (기존 파이프라인 동일)   │
                        │  ┌───────────────────▼─────────────────────┐  │
                        │  │        S3 Spooler Core (Phase 1)        │  │
                        │  │  Watcher → Stream Manager → S3          │  │
                        │  └─────────────────────────────────────────┘  │
                        └───────────────────────────────────────────────┘
```

### 6.3 설계 방향

**핵심 원칙**: 수신 방식만 확장하고, 스풀 디렉토리 이후의 파이프라인은 변경하지 않는다.

| 항목 | 내용 |
|------|------|
| 수신 방식 | HTTP multipart/form-data POST (또는 chunked PUT) |
| 파일 저장 | `filename_codec.encode()`로 스풀 파일명 생성 후 spool_dir에 atomic write |
| 인증 | mTLS (클라이언트 인증서) 또는 API Key 헤더 (상세 검토 필요) |
| 구현체 후보 | `aiohttp` 서버 (기존 asyncio 루프에 통합) |
| 설정 추가 | `http_enabled`, `http_port`, `http_host`, `http_auth_*` |

### 6.4 Phase 2 진입 조건 체크리스트

- [ ] FR-01~FR-05 로컬 단위 테스트 전체 통과
- [ ] 실제 Core Device에서 Stream Manager 연동 검증 완료
- [ ] 기능정의서 Phase 2 항목 별도 작성

> A-07(Pattern 2 마이그레이션)은 **구현 완료**되어 현재 운영 설계의 기본이다. 상세는 아래 참고.

### Pattern 2 (S3ExportTaskDefinition) — 구현됨

> 2026-06-01 SDK 소스 직접 확인으로 설계 확정, 이후 `stream_client.py`에 구현 완료.

`S3ExportUploader`가 파일 내용을 직접 전송하지 않고, `S3ExportTaskDefinition`을 직렬화해
append한다. Stream Manager가 해당 파일을 직접 읽어 S3 멀티파트 업로드하므로 **파일 크기
제한이 없고 per-file S3 키 제어가 가능**하다.

```python
# stream_client.py (S3ExportUploader.append_file)
task = S3ExportTaskDefinition(
    input_url=f"file://{spool_file_path.absolute()}",  # SM이 파일을 직접 읽음
    bucket=config.s3_bucket,                            # SpoolerConfig.s3_bucket
    key=s3_key,                                         # per-file S3 키 완전 제어
)
seq = client.append_message(
    stream_name,
    Util.validate_and_serialize_to_json_bytes(task),   # JSON 직렬화 (cbor2 아님)
)
# → SM이 파일 읽기 → S3 멀티파트 업로드 (5MB 이상 자동 멀티파트)
# → 완료 시 status_stream에 StatusMessage(Success/Failure) 기록
```

**스트림 설정** (관리자가 사전 생성 — [08](08-S3-스트림-관리-가이드.md) §2.2):
`S3ExportTaskExecutorConfig`에는 `s3_bucket`/`key_prefix`가 없으며, 버킷·키는 메시지마다
`S3ExportTaskDefinition`으로 지정한다. `size_threshold_for_multipart_upload_bytes=5_242_880`로
5MB 초과 시 자동 멀티파트, `status_config`로 완료 상태 스트림을 지정한다.

**파일 라이프사이클**:
```
1. 파일 감지 → append_file(task_definition) → seq_num 획득 (S3ExportUploader)
2. status_stream에서 업로드 완료 확인 (check_upload_status)
3. Status.Success 또는 상태 스트림 미설정 → 스풀 파일 삭제 (S3SpoolerClient)
4. 실패 → ERROR 로그, 파일 보존 (보존 기간까지 재시도 허용)
```
