## 1. 목적 및 범위

### 1.1 목적
AWS Greengrass 기반 IoT 디바이스에서 여러 로컬 프로세스가 생성한 파일을 안정적으로 AWS S3에 업로드하기 위한 스풀러 컴포넌트를 제공한다.

### 1.2 범위
- **포함**: 스풀 디렉토리 감시, 파일명 기반 라우팅, Stream Manager 전송, 자동 공간 관리, 파일명 인코딩 CLI 도구
- **제외**: Stream Manager 스트림 생성/설정, S3 버킷 생성, IAM 권한 설정

---

## 2. 기능 요구사항

### FR-01: 파일 감시 및 전송

**설명**: 스풀 디렉토리에 새 파일이 생성되면 자동으로 감지하고 Stream Manager에 전달한다.

**상세 조건**:
- 파일이 완전히 기록된 후 전송을 시작한다 (클라이언트는 atomic rename 사용 권고 — [04-인터페이스설계서](04-인터페이스설계서.md) §1.2 참고)
- Spooler 재시작 시 스풀 디렉토리에 남아 있는 파일도 처리한다
- 전송 실패 시 파일을 삭제하지 않고 재시도를 허용한다

**검증 기준**:

| 파일 크기 | 전달 완료 + 삭제까지 | 근거 |
|----------|---------------------|------|
| ≤ 1 MB | **1초 이내** | [research/01](research/01-latency-requirements-rationale.md): watchdog 50ms + SDK 10ms |
| ≤ 64 MB | **10초 이내** | [research/01](research/01-latency-requirements-rationale.md): 64MB SDK 전송 ~7ms + 여유 |

> **이전 기준(30초)과의 차이**: 이전 기준은 Stream Manager → S3 전체 파이프라인을 포함한 추정값이었다. 이 컴포넌트의 책임은 "Stream Manager에 메시지 append 완료"까지이므로 NFR-02와 일치하도록 수정한다.

---

### FR-02: 파일명 기반 라우팅

**설명**: 파일명 하나만으로 전송 목적지(스트림, S3 경로, 원본 파일명)를 결정한다.

**파일명 형식**:
```
{stream_id}__{s3_key_encoded}
```

**필드 정의 및 명명 규칙**:

| 필드 | 규칙 | 설명 |
|------|------|------|
| `stream_id` | regex: `^[a-z][a-z0-9\-_]*$` (최대 64자) | Stream Manager 스트림 이름. 형식: `{process_name}` (고정). 소문자로 시작, 소문자·숫자·하이픈·언더스코어만 허용. 동적 구분은 S3 키에서 처리. 예: `telemetry`, `device-001`, `camera-front` |
| `s3_key_encoded` | 슬래시를 느낌표(!)로 치환 | S3 키 경로의 인코딩 값 (파일명 포함) |

**stream_id 명명 규칙 근거**:
- 소문자 전용: S3 버킷명 규칙(소문자만), 대소문자 혼용 오류 예방
- 하이픈 권장: 명시적 단어 구분, 프로세스명과 sequence 분리 용이
- 64자 제한: 10~100ms 범위의 SM 메시지 크기 여유 확보
- 클라이언트 계약: 스풀러와 클라이언트 간 유일한 네이밍 계약

**클라이언트 부담 완화**:  
클라이언트 프로세스가 슬래시 치환을 직접 구현하지 않아도 되도록 세 가지 방법을 제공한다.

```bash
# 방법 1: CLI 도구 (모든 언어에서 subprocess 호출 가능)
python3 -m spooler encode telemetry "data/device-1/reading.json"
# → telemetry__data!device-1!reading.json

# 방법 2: Python 클라이언트 라이브러리
from spooler.filename_codec import encode
spool_name = encode("telemetry", "data/device-1/reading.json")

# 방법 3: Shell 스크립트 (순수 bash/sh)
S3_KEY="data/device-1/reading.json"
ENCODED=$(printf '%s' "$S3_KEY" | tr '/' '!')
SPOOL_NAME="telemetry__${ENCODED}"
```

상세 사용법은 [07-클라이언트-통합-가이드](07-클라이언트-통합-가이드.md)를 참고한다.

**검증 기준**: `encode()`/`decode()` 왕복 변환 시 원본 값이 복원된다. 한글·특수문자 S3 키도 지원된다.

### FR-03: 다중 클라이언트 지원

**설명**: 여러 프로세스가 동시에 스풀 디렉토리에 파일을 저장할 수 있다.

**상세 조건**:
- 각 클라이언트는 독립적으로 다른 스트림/경로를 지정할 수 있다
- 동일 스트림에 여러 파일을 동시에 전송할 수 있다
- 파일 간 전송 순서는 FR-06에 따른다

---

### FR-04: 보존 기간 기반 자동 삭제

**설명**: 설정된 보존 기간을 초과한 파일은 공간 여유와 무관하게 삭제한다.

**기본값**: 120시간 (5일) | **기준**: 파일의 마지막 수정 시간(mtime)

**정량 근거**: 건설기계 TGU 운영 특성상 주말·연휴 동안 네트워크 단절이 지속될 수 있어, 주말 휴무를 포함한 5일치 미전송 파일을 보존한다. 지연 요구사항 분석은 [research/02](research/02-retention-period-rationale.md)를 참고한다.

**주의**: FR-05 용량 한도가 먼저 발동하면 보존 기간보다 일찍 파일이 삭제될 수 있다. 두 정책은 **상호 보완**하며, 어느 조건이든 먼저 충족되면 삭제한다.

**검증 기준**: mtime이 보존 기간보다 오래된 파일은 다음 정리 사이클에서 삭제된다.

---

### FR-05: 용량 기반 자동 삭제

**설명**: 스풀 디렉토리 전체 크기가 설정 한도를 초과하면 오래된 파일부터 삭제한다.

**기본값**: **900 MB** | **삭제 순서**: mtime 오름차순 (가장 오래된 파일 먼저)

**정량 근거**: 스풀러 전용 파티션 1GB 기준. 파일시스템 오버헤드(20MB), 안전 여유 5%(50MB), 컴포넌트 아티팩트(5MB)를 제외하면 약 949MB가 가용하다. 900MB는 추가 여유 ~50MB를 포함한 권장값이다. 상세 계산은 [research/03 v1.1](research/03-spool-sizing-rationale.md)를 참고한다.

> **⚠️ v1.0 → v1.2 변경**: 기존 400MB 기본값은 스풀 파티션을 OS/Greengrass와 공유한다고 잘못 가정한 결과였다. 전용 파티션 확인으로 900MB로 상향 수정.

**파일 크기 제약 연계**:
- Pattern 2(FR-07)는 파일 크기 제한이 없으나, 스풀 한도(`max_spool_size_mb`)는 동시에 머무를 수 있는 미전송 파일 총량을 제한한다
- 단일 파일이 스풀 한도를 초과하면 정리 사이클에서 즉시 삭제될 수 있으므로, 예상 최대 파일 크기보다 충분히 큰 한도를 설정한다

**검증 기준**: 총 파일 크기가 한도를 초과하면 한도 이하가 될 때까지 mtime 오름차순으로 삭제된다.

---

### FR-06: 파일 처리 순서

**설명**: 스풀 디렉토리에 여러 파일이 있을 때 처리 순서를 정의한다.

**결정: 전역 순차 처리**

- 스풀 파일은 **mtime 오름차순** (오래된 것 먼저)으로 **하나씩 직렬 처리**한다
- 같은 스트림 내 데이터 순서가 S3에서 보장된다
- 다른 스트림 간에는 하나의 큐를 공유하므로 순서가 혼재될 수 있다

**근거**: 단순 구현으로 버그 가능성을 낮추고, 시계열 데이터 순서를 전역 보장한다. 상세 분석은 [research/05](research/05-processing-model-rationale.md)를 참고한다.

**향후 전환 조건**: 다중 스트림 환경에서 NFR-02를 초과하는 지연이 실측되면 스트림별 순차 처리로 전환을 검토한다.

---

### FR-07: 대용량 파일 전송 (Pattern 2)

**설계**: `S3ExportTaskDefinition` 패턴(Pattern 2)을 사용한다. Stream Manager가 스풀 파일을 직접 읽어 S3 멀티파트 업로드를 자동 처리하므로 **실질적 파일 크기 제한이 없다**. (`append_message(data=raw_bytes)` 방식의 64MB 한도는 적용되지 않는다.)

**구현 (`stream_client.py`, 구현 완료)**:
- 스풀 파일의 로컬 경로를 `S3ExportTaskDefinition.input_url`로 등록 (`S3ExportUploader`)
- Stream Manager가 파일을 직접 읽어 S3 멀티파트 업로드 처리 (5MB 초과 시 자동 멀티파트)
- per-file S3 키 직접 제어 (`bucket`, `key` 파라미터) — 파일명의 `s3_key`가 실제 S3 키가 됨
- 클라이언트 측 청크 분할 불필요

**검증 기준**:
- 임의 크기 파일(1MB~수GB)이 단일 S3 오브젝트로 업로드됨
- 지정한 `s3_key`가 실제 S3 오브젝트 키로 사용됨
- 업로드 완료 확인 후 스풀 파일이 삭제됨

## 4. 제약 사항

- Stream Manager S3 Export 스트림은 사전에 생성되어 있어야 한다 ([08-S3-스트림-관리-가이드](08-S3-스트림-관리-가이드.md) 참고)
- stream_id는 명명 규칙을 준수해야 한다 (FR-02 참고: regex `^[a-z][a-z0-9\-_]*$`, 최대 64자)
  - 이를 위반하면 `ValueError` 발생
- 미등록 stream_id도 사용 가능하나, 동일 이름의 Stream Manager 스트림이 존재해야 전송에 성공한다 (없으면 전송 실패, 파일 보존)
- 스풀 파티션(1GB 전용)과 Greengrass 파티션은 별도이어야 한다
- S3 오브젝트 키는 파일명의 `s3_key`(슬래시→`!` 인코딩 해제값)가 직접 결정한다 (Pattern 2, FR-07)
