## 2. 분석

### IoT 데이터 특성

- **시계열 순서 보장 필요**: 같은 스트림 내 파일은 생성 순서대로 S3에 기록되어야 데이터 연속성이 유지된다.
- Stream Manager는 `append_message` 순서대로 S3에 저장한다.
- 스트림이 다르면 순서가 독립적이므로 병렬 처리해도 데이터 정합성에 영향 없다.

### 처리량 vs 복잡성

| 요소 | 전역 순차 | 스트림별 순차 |
|------|-----------|-------------|
| 구현 복잡도 | 낮음 | 중간 (스트림별 큐 관리) |
| 단일 스트림 처리량 | 동일 | 동일 |
| 다중 스트림 처리량 | 낮음 | 높음 |
| 버그 가능성 | 낮음 | 중간 (큐 누수, 데드락) |
| 메모리 사용 | 낮음 | 스트림 수 × 큐 크기 |

### 현재 예상 사용 패턴 분석

요구사항에서 "여러 클라이언트 프로세스, 각각 다른 스트림 또는 같은 스트림"을 명시.

- 단일 스트림으로만 사용: 전역 순차 = 스트림별 순차 (차이 없음)
- 다중 스트림: 스트림별 순차가 처리량 유리
- 그러나 처리량 병목이 실제 발생하는지는 확인 필요

---

## 3. 결정: 전역 순차 (단계 1)

**이유**:
1. 현재 단계(Phase 1)에서 처리량 병목이 확인되지 않았다
2. 구현이 단순하여 버그 가능성이 낮다
3. 데이터 순서 보장이 최우선이며, 전역 순차는 이를 완전히 보장한다
4. 향후 스트림별 순차로 전환 시 `SpoolWatcher` 내 큐 구조만 변경하면 된다 (인터페이스 불변)

**mtime 기준 정렬**: 큐 진입 순서는 `path.stat().st_mtime` 오름차순 (오래된 것 먼저).

## 4. 스트림별 순차 전환 조건

아래 조건이 만족될 때 스트림별 순차 처리(모델 C)로 전환을 검토한다:

- [ ] 다중 스트림 운영 환경에서 처리 지연이 NFR-02(1초)를 초과하는 실측값 확인
- [ ] 스트림 수가 상시 3개 이상

전환 구현 개요:
```python
# 현재: 단일 큐
self._queue: asyncio.Queue[Path]

# 전환 후: 스트림별 큐 + 스트림별 워커
self._queues: dict[str, asyncio.Queue[Path]]
self._workers: dict[str, asyncio.Task]
```
